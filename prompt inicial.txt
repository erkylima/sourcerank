Cenários essenciais (starter só em init ou troca de linguagem local; nunca sobrepor texto já existente)

A) Novo desafio, DB vazio, relay sem snapshot

Ação: Seed DB/starter → aplica linguagem+conteúdo numa transação, envia bootstrap se relay vazio.
Risco: nenhum; é o caso canônico.
B) Novo desafio, relay tem snapshot (outro cliente já editou)

Ação: Aplicar snapshot, não semear starter nem DB.
Risco: sem starter; correto, preserva estado real.
C) Troca de linguagem local (usuário decide)

Conteúdo vazio/igual ao starter antigo: pode semear starter da nova linguagem e meta.language juntos.
Conteúdo já editado: não semear starter, só mudar meta.language; mantém texto.
Risco evitado: não apagar trabalho ao trocar linguagem.
D) Troca de linguagem recebida via CRDT (outro cliente mudou)

Ação: Atualizar language local; não semear starter, não tocar no texto.
Risco evitado: não pisar no texto do outro.
E) Reload com DB contendo conteúdo+linguagem; relay silencioso

Ação: Seed com DB (conteúdo+linguagem) e enviar bootstrap uma vez.
Risco: nenhum.
F) Navegar para desafio já salvo

Ação: Carregar via DB/relay; se houver snapshot, não semear; se vazio, semear starter/DB uma vez.
Risco: não sobrescrever conteúdo salvo.
G) Cliente antigo (não publica meta.language)

Ação: Novo cliente não re-seeda starter quando recebe linguagem; apenas alinha language.
Risco mitigado: não gerar overwrite por cliente legado.
H) Alta latência / envio fora de ordem

Ação: Não semear starter se o texto já diverge do starter; language e starter só na mesma transação quando permitido.
Risco mitigado: evitar starter tardio sobrescrevendo edição recente.
